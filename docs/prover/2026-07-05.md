## Triage

## Opening assessment

This is a mature, unusually disciplined spec — the credibility layer (§A/§B), the missing/measured/measured-zero
trichotomy (§E), and the one-surface-one-name stem-label partition (INV-STEMNAME-*) are all strong, and the
cross-axis composition (player × view in §B.14, remembered view in §B.15, completeness × mode in §E.3, design
tokens × view/persistence/viewport in §I.10) is exactly the product-space where the hard bugs hide — and it's
covered explicitly. The spec is close to gate-ready.

The one thing standing between it and a clean 1.0 is **not a hole in reasoning — it's an unresolved boundary**:
§D (reference/compare) is ~600 lines in a half-deferred, half-shipping superposition, and that superposition
now disagrees three ways (spec says descriptive §D ships in 0.9 · the completeness gate marks those surfaces
DEFERRED · Alexander expects them shipped). For a 1.0 gate whose whole point is "ship whole, prove whole," a
surface that is supposed to ship but the anti-regression net doesn't guard is the top item. Everything else is
smaller: a couple of internal-consistency drifts in the label-set enumeration, and two spec facts (spacing
tokens, significance time-coverage) that describe intended-not-shipped behaviour and should be marked as such
so the spec doesn't read as asserting them current.

**Confidence: needs another iteration** — one focused §D boundary decision + three small consistency fixes,
not a rework.

## Findings

### F1 — The §D ship/defer boundary disagrees three ways; the completeness gate doesn't guard a surface that is meant to ship

> "The **descriptive** rows below … ship in **0.9**: they need no aspiration mapping, only the measured
> fingerprints." — §D.10.1, Scope split

> `"refRead": {"condition": "DEFERRED", "gated_by": None}` / `"webPanel": {"condition": "DEFERRED", …}` —
> `tests/test_completeness_gate.py:88-89` (verified by command, s54)

The spec says the descriptive reference read + web panel ship in 0.9; the completeness gate
(`test_completeness_gate.py:86-89`) marks `refRead` and `webPanel` `DEFERRED` / `gated_by: None`, so they are
**excluded from the whole-artifact gate**; and `NEXT_STEPS.md` records that Alexander expects the surviving §D
(nearest-match similarity) **shipped**. So the anti-regression net — the mechanical "every user-facing surface
present + non-empty" gate that is the project's core discipline — does not cover a surface the spec+owner say
ships. A regression that blanks the reference read in a build that has reference directions would pass every gate.

Resolve the boundary explicitly, in ONE place, before 1.0: decide the 1.0 §D scope (recommend: **descriptive
nearest-match §D + §F own-library SHIP; aim/re-flavouring stays cut**), then make the three agree — (a) SPEC
§D marks each surface `SHIPS-1.0` vs `DEFERRED-post-1.0` on one line; (b) `USER_SURFACES` gates the shipping
ones with a real `gated_by` test and a render condition ("reference directions defined"); (c) a matrix row per
shipping §D surface at L3-BROWSER. This is the pass-1 gate blocker.

`must-fix · boundary-issue (composition)`

### F2 — Multiple "⟨pending Alexander's confirm to cut for 1.0 vs rebuild⟩" left open on the aim/aspiration path

> "⟨pending Alexander's confirm to cut for 1.0 vs rebuild the mapping input⟩" — §D.6, and again §D.10.1
> DEFERRED note, and §B.7 arc-lead partition ⟨pending Alexander's confirm⟩

The aspiration-mapping input surface (the aim picker, ⟨DECIDE D-2⟩) was **excised in 0.9.15**, which strands a
family of invariants that reference it — D-INV-4 (mapping is yours), D-INV-13 (delete cascade), D-INV-15/17
(re-flavour transparency/order), D-INV-18 (add/remove symmetry), the aim-glyph parts of D-INV-28 — as

text under a single clearly-labelled `## §D — POST-1.0 (aspiration mapping, re-flavouring)` heading so the

retires all three markers. This is Alexander's call — it's the one genuine decision in this pass.

`must-fix · missing-scenario (state-space)` — *needs author input (scope), not inspection.*

### F3 — The "displayed lane label is EXACTLY one of {…}" enumeration is stale vs the near-silent-ID rule

> "The displayed lane label is EXACTLY one of: `bass, drums, kick, perc, hats, lead, melody, chord, pad, mid,
> high, near-silent, not measured`." — §B.7 (line ~356)

> "an **IDENTIFIED near-silent label** … the mapped real project-track name … ELSE a frequency-band descriptor
> (`low`/`low-mid`/`mid`/`high` …) — always suffixed ` (near-silent)`; **NEVER a bare `near-silent`**" —
> §B.7 INV-STEMNAME-ALL (line ~374)

Two rules in the same section describe the label set differently. The older enumeration says a lane can read a
bare `near-silent`; the newer s45/s46 rule (INV-STEMNAME-NEARSILENT-ID) says it must read `⟨track-or-band⟩
(near-silent)` and **never** a bare `near-silent`, and introduces band words `low`/`low-mid` not in the
enumeration. A test that asserts membership in the stale "EXACTLY one of" set would either reject a correct
`Guitar (near-silent)` label or accept a forbidden bare `near-silent` — the matrix row is derived from a set
the doc contradicts one paragraph later.

Rewrite the "EXACTLY one of" enumeration to match INV-STEMNAME-NEARSILENT-ID: significant → a character label
from the character set; near-silent → `⟨identifying word⟩ (near-silent)` (never bare); unknown → `not measured`.
State the near-silent form as a *pattern*, not a fixed-word member. Then re-pin the label-set matrix row to it.

`should-clarify · internal-conflict (consistency)`

### F4 — When does a standalone `kick` lane label ever appear, given `drums → "drums"`?

> "a `drums` stem is **"drums"** (not "kick" — kick is a drum-breakdown sub-part)" — §B.7
> …yet `kick` is listed as a valid displayed lane label — §B.7 enumeration + DS-INV-7b stem colour `kick`

§B.7 trusts the low-end families and renders a `drums` stem as "drums," explicitly *not* "kick." G12 produces
`kick` only for a low+percussive stem — but a low+percussive stem separated by Demucs *is* the drums family,
which §B.7 routes to "drums." So the paths that could emit a standalone `kick` lane label appear to be closed,
yet `kick` is still enumerated as a lane label (and carries its own stem colour). Either there's a real path
(a non-drums stem typed low+perc) that should be named, or `kick` is a phantom member of the lane-label set
that a membership test will never exercise.

State the one path that yields a `kick` lane label (an example stem), or drop `kick` from the lane-label
enumeration and keep it only where it's real (the drum-hit breakdown sub-parts, §B.11). A membership test
should never carry an unreachable member.

`should-clarify · over-specific (abstraction)`

### F5 — DS-INV-9 (spacing tokens) is specified as system but ships as raw literals — mark it, don't assert it

> "**Spacing split (DS-INV-9):** two roles — `--gap 8/12/16` (within a group) and `--rhythm 28/44` (between
> sections)." — §I.2

`NEXT_STEPS.md` records (correctly) that the widget still uses raw `gap:` literals and `:root` defines neither
token — so §I.2 states a token system as part of the design system while the shipped code has none. This is
already logged as an honest not-built matrix row, so it isn't hidden; the issue is that §I reads as **current
system spec**, and the §I.9 machine-checked list could be read to imply the tokens exist. For the 1.0 gate,
either build the two tokens (mechanical: add to `:root`, snap the literals) or move DS-INV-9 to a §I.11
"post-1.0" line so §I doesn't assert an unshipped token set as part of the shipped system.

Recommend: **build it** — it's a one-file mechanical snap like the radii/motion tokens already done, and it
closes the last design-token drift before the MAJOR. If deferred, mark it `POST-1.0` inline in §I.2.

`worth-considering · unenforceable-promise (discharge)`

### F6 — Architecture pins are file-level, not `file:line`; the "one fact → one node → owning line" chain is loose for the gate

> "N8 | The credibility layer … | `build_widget.py` (claim assembly), `render_spec.py`, `track_analyzer.py`"
> — `ARCHITECTURE.md`, L3 table (no line numbers)

The pipeline's step-3 rule is "nodes pinned to owning `file:line`." The arch doc pins to files (and some nodes
to 2–3 files). For most nodes that's fine, but N8/N12/N16/N17 all point at the 269 KB `build_widget.py`

gate — a fact could be double-owned across N12/N15/N16 inside that one file and nothing would catch it. Not a
1.0 blocker, but it weakens pass-2's ability to check fact-ownership.

Add owning `file:line` (or a stable function name) to the sub-nodes of `build_widget.py` (N12–N17) — the
matrix already references functions like `_web_panel_html`, `mask_collision_freq`; lift those into the arch
table so pass-2 can check one-fact-one-node mechanically.

`worth-considering · boundary-issue (composition)`
